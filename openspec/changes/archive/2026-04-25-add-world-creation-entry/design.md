## Context

当前前端已经完成 Next.js App Router 工程底座，并在应用根路径呈现“烟雨江南”水墨工作台。该工作台代表“已有世界”的运行与展示状态，但用户还没有从空状态或工作台主动创建世界的入口。

本变更只覆盖首版前端入口：独立创建世界页面、左侧导航入口、表单字段、世界类型选择、实时预览和创建后进入现有工作台的前端状态接入。后端持久化、世界运行、角色生成和 Agent 工作流不在本阶段实现。

受影响文件和模块预计包括：

- `src/app/page.tsx`：工作台入口可能需要接收创建后的前端状态。
- `src/app/worlds/new/page.tsx`：新增创建世界页面。
- `src/components/dashboard/ink-dashboard.tsx`：新增“新建世界”入口，允许顶部世界信息被创建数据覆盖。
- `src/components/world-creation/**`：新增创建页表单、类型选择卡和预览组件。
- `src/lib/dashboard-data.ts`：保留默认 mock 数据，并可能抽取创建后映射逻辑。
- `src/types/**`：新增或扩展世界创建表单、世界类型和工作台展示数据类型。
- `src/app/globals.css`：延续水墨主题，增加创建页布局、表单和预览样式。

## Goals / Non-Goals

**Goals:**

- 提供 `/worlds/new` 独立页面作为创建世界的核心入口。
- 在现有侧栏中提供清晰的“新建世界”入口。
- 用单页分区表单承载首版六个字段，避免过早引入多步向导。
- 用选择卡展示六个世界类型，并支持“自定义”类型名称输入。
- 在右侧实时预览用户填写的世界雏形。
- 首版点击“创建世界”后进入现有工作台，并使用前端 mock 或本地状态展示新世界名称、简介和类型标签。
- 保持现有水墨工作台视觉基调、响应式可读性和前端验证命令可用。

**Non-Goals:**

- 不实现登录、权限、世界列表或多世界切换系统。
- 不新增后端 API、数据库表、持久化模型或迁移脚本。
- 不触发角色生成、阵营生成、世界运行、日志生成或 LLM/Agent 调用。
- 不设计完整 onboarding、多步向导、模板市场或 AI 辅助填充。
- 不改变已有工作台核心信息架构和右侧栏模块。

## Decisions

1. 创建世界使用独立页面 `/worlds/new`，而不是弹窗或工作台内嵌表单。

   理由：背景简介、世界规则和初始冲突都属于长文本创作输入，需要足够空间和稳定布局。弹窗会挤压文本编辑体验；内嵌在工作台会混淆“创建新世界”和“管理当前世界”。独立页面更适合后续扩展草稿、模板、AI 辅助生成和校验。

   备选方案：在工作台顶部“世界设定”按钮中加入创建弹窗。该方案入口更近，但字段密度高、移动端体验差，也不利于空状态承接。

2. 首版采用“单页分区 + 右侧实时预览”，暂不做多步向导。

   理由：当前字段只有六个，用户需要在填写时整体感知世界底色和冲突种子。单页可以减少流程摩擦，右侧预览能即时反馈。多步向导适合后续字段增多、加入角色/势力/规则模板后再引入。

   备选方案：按“基础设定 → 规则 → 冲突 → 确认”拆成多步。该方案仪式感更强，但首版实现和验证成本更高，也容易让用户在前几步缺少全局上下文。

3. 世界类型使用卡片选择，选项固定为“宫廷权谋、魔法学院、末日避难所、小镇悬疑、修仙宗门、自定义”。

   理由：世界类型是创作心智的起点，卡片可以承载关键词和图标，帮助用户快速理解类型差异。普通下拉节省空间，但弱化了创作启动感，也难以展示类型解释。

   备选方案：自由文本输入世界类型。该方案灵活，但缺少引导，无法保证首版预览和后续模板接入的稳定性。

4. “自定义”类型必须展开自定义类型名称输入。

   理由：如果只记录类型为“自定义”，工作台标签和后续世界规则推断都会缺少具体语义。单独输入可以保持数据可读，并为后续模板或后端模型接入留下明确字段。

5. 创建后的首版状态使用前端 mock 或本地状态，不做持久化。

   理由：本阶段目标是打通入口体验，不应该提前设计后端世界模型。可以先用 URL 跳转、组件状态、sessionStorage/localStorage 或轻量客户端状态把创建数据带入工作台展示。实现时应选择与现有 App Router 和组件边界最简单的一种方式，并保持数据结构易于替换为后端响应。

   备选方案：直接新增后端创建世界 API。该方案更接近真实产品，但会引入数据库、接口契约、错误处理和 Agent 边界，超出首版入口范围。

6. 视觉风格延续现有水墨工作台，但创建页不照搬三栏信息密度。

   理由：创建页是输入型界面，应更安静、聚焦表单和预览。保留深色侧栏、宣纸底、细线边框、低饱和色和图标按钮即可，不需要复用工作台中的运行状态、事件列表和右侧情报模块。

## Risks / Trade-offs

- [Risk] 使用本地状态后刷新页面可能丢失刚创建的世界信息。→ Mitigation：首版在验收中明确“不保证持久化”；如使用 sessionStorage/localStorage，也仅作为临时体验，不作为真实数据模型。
- [Risk] 创建页表单长文本过多，移动端可能显得拥挤。→ Mitigation：窄屏下改为单列布局，预览区域移动到表单后方，并限制输入区最小高度和文字换行。
- [Risk] 左侧导航新增入口可能影响现有工作台布局高度。→ Mitigation：将“新建世界”作为品牌区下方的独立按钮或导航首项，保持原导航项尺寸稳定。
- [Risk] 自定义类型输入为空时创建，会产生不可读标签。→ Mitigation：创建按钮在必填字段不满足时禁用，或提交时显示内联错误。
- [Risk] 过早把“创建世界”逻辑写死在 dashboard mock 数据中。→ Mitigation：新增轻量类型和映射函数，把创建表单数据转换为工作台可展示世界信息，后续替换后端响应时减少改动。
- [Risk] 前端直接拼接复杂世界规则会越界到业务运行逻辑。→ Mitigation：首版仅展示用户输入，不解析、不推导、不生成世界运行状态。

## Migration Plan

1. 新增创建页、创建表单组件、类型选择数据和预览组件。
2. 在现有工作台侧栏加入“新建世界”入口，链接到 `/worlds/new`。
3. 实现创建后的前端临时状态传递，让工作台顶部世界信息可展示用户输入。
4. 保留默认“烟雨江南”mock 数据作为没有创建状态时的回退。
5. 运行 lint/类型检查和浏览器截图验证，覆盖工作台与创建页的桌面和窄屏视口。

回滚策略：删除 `/worlds/new` 页面和 `world-creation` 组件，移除新增类型与本地状态映射，恢复侧栏导航为原有条目，工作台继续固定展示 `dashboardData.world`。

## Open Questions

- 首版创建后是否使用 `sessionStorage`、`localStorage`，还是仅在客户端内存中保留到当前会话？实现前可按最小可用方案决定。
- 创建按钮是否要求所有六个字段必填，还是只要求世界名称、世界类型、背景简介和初始冲突必填？建议首版至少校验名称、类型和自定义类型名称。
- `/` 根路径未来是否应根据“有无已创建世界”自动跳转到 `/worlds/new` 或工作台？本变更先保留现有根路径工作台，待世界列表/持久化出现后再定义。
