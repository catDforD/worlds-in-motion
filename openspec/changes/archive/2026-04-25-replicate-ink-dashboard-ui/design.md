## Context

当前仓库已经完成 Next.js、TypeScript、Tailwind CSS 与 shadcn/ui 的前端工程初始化，首页仍是最小验证页面。用户希望基于上传的参考图，将织世录主界面具体设计为一个完整复刻的水墨古风工作台：左侧深色导航、顶部世界标题与山水横幅、中部运行概览和内容卡片、右侧状态栏、底部活跃角色横排。

本阶段是前端静态界面变更，不接入真实世界运行数据、不新增 API、不修改后端或 Agent 工作流。实现时需要使用 mock 数据承载截图中的信息密度，并用浏览器截图验证视觉结果。

受影响文件和模块预计包括：`src/app/page.tsx`、`src/app/globals.css`、新增的 `src/components/dashboard/**`、`src/types/**`、`src/lib/**`，以及可能放入 `public/` 或 `src/img/` 的视觉资源。

## Goals / Non-Goals

**Goals:**

- 复刻参考图的桌面端主工作台布局和模块层级。
- 建立水墨古风视觉系统：宣纸背景、墨色侧栏、细线卡片、山水横幅、低饱和色彩、中文字体层级和图标按钮。
- 用静态 mock 数据填充世界标题、标签、统计数字、运行状态、事件、故事章节、角色卡、紧张关系、待解秘密和近期趋势。
- 保持 Next.js 页面可构建、可类型检查、可 lint。
- 使用 Playwright 或等效浏览器截图检查桌面和窄屏结果，确认文本、卡片和侧栏没有明显重叠或溢出。

**Non-Goals:**

- 不实现真实导航路由、登录、世界设置弹窗或通知系统。
- 不接入 FastAPI、PostgreSQL、pgvector、LangGraph 或 LLM。
- 不实现真实沙盘运行、角色行动、关系计算、叙事生成或持久化。
- 不追求逐像素级还原截图中的所有绘画细节；界面结构、比例、视觉气质和信息密度必须高度一致。

## Decisions

1. 使用单页静态工作台替换当前最小首页。

   理由：用户目标是复刻上传图中的完整主界面，当前阶段最直接的验收面是根路径。备选方案是新增 `/dashboard` 并保留空首页，但会让用户打开根路径时看不到目标成果，也不符合“基于现在项目框架设计前端界面”的意图。

2. 将界面拆为数据配置和展示组件。

   计划将 mock 数据集中到页面文件附近或 `src/lib/dashboard-data.ts`，并把侧栏、顶部横幅、统计条、运行状态、事件列表、故事卡、角色卡和右侧栏拆成组件。这样既能保持静态实现简单，又避免 `page.tsx` 变成难以维护的长模板。备选方案是在单文件内完成所有 JSX，但后续迭代成本更高。

3. 使用 CSS/Tailwind 构建水墨质感，优先复用已有资源和浏览器能力。

   宣纸底、边框、阴影、水墨渐层、卡片纹理和侧栏质感优先通过 CSS 变量、伪元素、渐变、透明度和背景图组合实现。头像、山水横幅和章节插图可使用已有参考资源、可用的本地图片或后续生成的位图资产。备选方案是把整张截图作为背景图并覆盖热区，但这会失去真实布局、响应式和文本可维护性。

4. 桌面端采用固定信息架构，窄屏采用可滚动降级。

   参考图是宽屏仪表盘，桌面端应使用左侧固定宽度导航、主内容区和右侧信息栏三栏布局。窄屏时优先保持内容可读：侧栏转为顶部/横向导航或折叠式导航，右侧信息栏下移，角色列表横向滚动。备选方案是强制缩放整屏，但会造成文本不可读。

5. 使用 lucide-react 图标表达导航和操作。

   项目已安装 lucide-react，适合侧栏、顶部按钮、统计项和趋势箭头。备选方案是手写 SVG 或纯文字按钮，但一致性和可维护性较差。

6. 视觉验收以可见结构和无破版为准。

   实现后需要运行 `npm run lint`、`npm run typecheck` 或 `npm run build`，并启动开发服务器用浏览器截图检查至少桌面宽屏和一个窄屏视口。验收重点是：主模块完整出现、文本不遮挡、卡片边界清晰、背景图不空白、滚动行为可用。

## Risks / Trade-offs

- [Risk] 参考图包含复杂山水绘画与角色头像，纯 CSS 无法完全复刻。→ Mitigation：将绘画类内容作为可替换位图资产处理，规格只要求资源稳定渲染和整体视觉一致。
- [Risk] 宽屏信息密度高，窄屏直接压缩会造成文字溢出。→ Mitigation：窄屏改为纵向布局和横向滚动列表，固定尺寸元素使用 `minmax`、`aspect-ratio`、`overflow-hidden` 和文本截断策略。
- [Risk] 古风视觉容易退化为单一米色/棕色调。→ Mitigation：引入墨黑、松绿、朱砂、灰青、纸白等功能色，并在 CSS 审查时确认不是单一色相。
- [Risk] 静态 mock 数据可能被误认为真实业务状态。→ Mitigation：代码中保持 mock 数据集中命名，界面不展示“示例说明”类文本，后续业务规格再替换数据来源。
- [Risk] 大量阴影、纹理或图片可能影响首屏性能。→ Mitigation：限制图片数量和尺寸，优先使用 Next/Image 或合理压缩的本地资源，避免全屏高分辨率透明叠层堆叠。
- [Risk] 使用外部图片存在版权或网络稳定性问题。→ Mitigation：实现阶段优先使用本地资源或可生成/可授权资产，不依赖远程图片 URL。

## Migration Plan

1. 在现有根页面中引入静态工作台布局，保持 App Router 结构不变。
2. 建立 dashboard 组件和 mock 数据，确保命名符合 TypeScript 规范。
3. 扩展全局样式或局部样式，加入水墨主题变量、背景纹理和响应式布局规则。
4. 接入本地视觉资源，并确认图片加载路径在 Next.js 构建中可用。
5. 运行 lint、类型检查或构建。
6. 启动本地开发服务器并用浏览器截图检查桌面和窄屏视口。

回滚策略：revert 本次变更，或删除新增 dashboard 组件、mock 数据、视觉资源，并恢复 `src/app/page.tsx` 与 `src/app/globals.css` 到前端基座初始化后的最小页面。
