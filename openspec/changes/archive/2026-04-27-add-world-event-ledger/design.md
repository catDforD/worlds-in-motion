## Context

根路径工作台已经具备水墨三栏界面，并通过 `src/lib/dashboard-data.ts`、`src/types/dashboard.ts` 和 `src/components/dashboard/ink-dashboard.tsx` 展示世界标题、角色、关系、近期事件、最新章节、当前时间和运行状态。现有运行态内容仍以 mock 数据为主，按钮也没有真实状态切换。

本变更首版只做前端本地运行记录，约束是：不调用后端、不写数据库、不触发 Agent。它需要和已有本地世界创建信息、世界设定、内容种子共存，并避免角色/关系种子逻辑误改运行日志。

受影响文件和模块预计包括：

- `src/types/dashboard.ts` 或新增 `src/types/world-runtime.ts`：定义 `WorldRuntimeState`、事件、章节草稿和最新章节类型。
- `src/lib/dashboard-data.ts`：将运行态默认值和工作台展示数据组合逻辑拆清楚。
- 新增或扩展 `src/lib/world-runtime.ts`：封装 localStorage key、读取、写入、初始化、迁移和异常回退。
- `src/components/dashboard/ink-dashboard.tsx`：读取本地运行态，渲染当前时间、运行状态、近期事件、最新章节，并提供暂停/继续、推进一日、记录事件入口。

## Goals / Non-Goals

**Goals:**

- 提供可本地持久化的 `WorldRuntimeState`，覆盖当前世界时间、运行天数、暂停状态、事件列表、章节草稿和最新章节。
- 工作台运行态模块优先读取本地记录，并在无记录或记录损坏时使用现有默认展示。
- 让“暂停运行”成为真实状态切换，并持久化到 localStorage。
- 提供轻量手动入口，让用户推进一日并记录事件。
- 保持前端静态检查、类型检查和浏览器截图验收可通过。

**Non-Goals:**

- 不接入 FastAPI、PostgreSQL、LangGraph 或 Agent 自动运行。
- 不实现复杂时间历法、角色自动行动、事件因果推演或章节自动生成。
- 不改变现有创建世界流程和内容种子的数据契约。
- 不新增跨设备同步或多世界服务端隔离。

## Decisions

1. 使用 localStorage 作为首版持久化介质。

   理由：当前创建世界和内容种子已经以本地前端状态为主，localStorage 能满足首版“本地可维护”的目标，避免引入后端和权限问题。备选方案 IndexedDB 更适合大量日志和复杂查询，但当前事件列表规模较小，会增加实现和测试成本。

2. 将运行态读写封装到独立工具模块。

   理由：工作台组件应该负责交互和渲染，不直接散落 localStorage 读写、JSON 解析和默认值合并逻辑。独立模块也便于后续替换为后端 API 或 IndexedDB。备选方案是在组件内直接读写，短期代码更少，但会让状态迁移、异常回退和测试变得分散。

3. `WorldRuntimeState` 保持面向展示的最小结构。

   建议字段包括：`currentWorldDate`、`runDays`、`isPaused`、`events`、`chapterDraft`、`latestChapter`、`updatedAt`。事件记录包含 `id`、`date`、`title`、`summary`、`type`、`createdAt`；章节记录包含 `title`、`summary`、`tags`、`updatedAt`。首版不把角色、势力、地点关系强绑定进事件模型，避免前端拼接复杂业务规则。

4. 工作台展示采用“本地运行记录优先，默认数据兜底”。

   理由：用户没有写入记录时，首页仍需要保持完整、可读的水墨工作台；一旦本地记录存在，当前时间、运行天数、暂停状态、近期事件和最新章节必须反映用户维护的记录。内容种子仍只影响角色、关系和统计，不伪造运行日志。

5. “推进一日”和“记录事件”合并为轻量入口但保持可单独表达。

   理由：用户的最小手动维护动作通常是推进时间并补一条事件。实现上可以提供一个表单/弹层/内联区域，允许只推进日期、只记录事件，或推进后记录事件。它不触发 Agent，也不生成章节。

## Risks / Trade-offs

- [Risk] localStorage 数据可能损坏或来自旧版本结构 → Mitigation：读取时做安全解析、字段默认值合并和异常回退，不让页面崩溃。
- [Risk] 单浏览器本地记录无法跨设备同步 → Mitigation：在本期明确为前端本地首版，后续可把独立工具模块替换为后端运行记录 API。
- [Risk] 事件持续增加后影响渲染和存储 → Mitigation：工作台只展示近期事件，写入时可保留完整数组但渲染限制数量；后续再设计分页或归档。
- [Risk] 手动入口被误解为自动世界运行 → Mitigation：按钮文案和交互只表达“手动推进/记录”，实现层不得调用 Agent 或后端。
- [Risk] 本地内容种子和运行记录都影响工作台，可能产生数据覆盖 → Mitigation：明确模块边界，角色/关系/统计读取内容种子，时间/状态/事件/章节读取运行记录。

## Migration Plan

1. 新增运行态类型和 localStorage 工具，读取不到记录时返回默认运行态。
2. 改造工作台运行态模块，优先使用 `WorldRuntimeState` 派生展示数据。
3. 接入暂停/继续、推进一日、记录事件的交互和持久化。
4. 执行静态检查/类型检查，并用浏览器截图验证桌面与窄屏工作台仍可读。

回滚时删除新增运行态工具和交互入口，将工作台对应模块恢复为 `dashboard-data` 的默认 mock 展示；localStorage 中遗留记录可忽略，不影响页面加载。

## Open Questions

- 世界时间首版是否固定使用现有 mock 的中文日期格式，还是保存 ISO 日期并在展示层格式化。
- 记录事件入口是否需要支持事件类型枚举，还是首版只需要标题和摘要。
- 章节草稿是否在工作台内可编辑，还是只作为运行态结构预留并展示最新章节。
