## Context

当前工作台已经能从创建世界临时状态和世界设定本地状态中读取世界名称、背景和标签，但角色、势力、地点、关系与统计概览仍来自 `src/lib/dashboard-data.ts` 的静态 mock 数据。`src/lib/world-settings.ts` 已经提供了一个可复用模式：版本化 `localStorage` payload、解析校验、损坏数据回退、`storage` 事件和自定义事件订阅。

本次变更需要让用户在创建世界后继续补充“世界内容种子”，并让工作台的角色、紧张关系和统计概览反映这些用户数据。实现仍限定在前端本地状态，不接入 FastAPI、PostgreSQL、LangGraph 或 LLM 编排。

受影响文件和模块预计包括：
- `src/types/dashboard.ts`：必要时扩展工作台角色、关系或统计类型。
- `src/types/world-seed-assets.ts`：新增角色、势力、地点、关系种子的类型定义。
- `src/lib/world-seed-assets.ts`：新增本地存储 key、解析校验、默认值映射、保存和订阅逻辑。
- `src/lib/dashboard-data.ts`：继续作为默认 mock 和视觉资源来源。
- `src/components/dashboard/ink-dashboard.tsx`：新增内容种子编辑入口/面板，并将统计、角色和关系模块改为接收派生数据。
- `src/app/globals.css`：新增或调整内容种子编辑界面的响应式样式。
- `output/playwright/`：新增或更新用于验收内容种子联动的浏览器脚本。

## Goals / Non-Goals

**Goals:**
- 提供角色、势力、地点和关系种子的首版本地管理能力。
- 使用版本化 `localStorage` 保存种子数据，并在刷新后恢复。
- 在本地数据缺失、损坏或为空时保持工作台可读，继续使用现有 mock 作为回退。
- 让工作台统计概览、活跃角色和紧张关系优先读取当前世界的本地内容种子。
- 保持现有创建世界和世界设定编辑体验可用，并避免破坏水墨工作台布局。

**Non-Goals:**
- 不新增后端 API、数据库表、用户账户隔离或跨设备同步。
- 不触发 Agent 运行、LLM 生成、事件生成或故事章节生成。
- 不把世界规则、禁忌、核心矛盾自动转化为角色关系或事件。
- 不实现复杂实体图谱、搜索、批量导入导出或版本历史。

## Decisions

1. 使用独立的 `world-seed-assets` 本地存储模块，而不是把种子字段塞进 `world-settings`。
   - 理由：世界设定是核心背景与约束，内容种子是可增长的实体集合；拆分后更容易未来迁移到后端表或 Agent 输入。
   - 替代方案：复用 `WORLD_SETTINGS_STORAGE_KEY`。缺点是 payload 会变得过大且职责混杂，后续迁移成本更高。

2. 采用版本化 payload，例如 `worlds-in-motion.world-seed-assets.v1`。
   - 理由：延续 `world-settings` 的本地状态模式，便于解析校验、损坏回退和未来迁移。
   - 替代方案：为四类种子使用四个 localStorage key。缺点是事务性更差，订阅和清理逻辑更分散。

3. 在 UI 层使用同一个内容种子面板管理四类种子，并以轻量表单增删改。
   - 理由：当前产品仍处于纯前端体验验证阶段，一个面板可以降低路由和导航复杂度，也符合工作台内整理世界底稿的使用场景。
   - 替代方案：分别创建角色、势力、地点、关系页面。缺点是实现面过大，会提前引入信息架构负担。

4. 将工作台展示数据从静态常量改为“静态默认值 + 本地种子派生值”。
   - 理由：统计、角色和关系必须反映用户自己的世界，但运行状态、近期事件、秘密和趋势仍不应被伪造为运行结果。
   - 替代方案：保存种子后直接重写 `dashboardData`。缺点是模块常量不可表达用户状态，也不利于服务端渲染边界和测试。

5. 关系种子引用参与方名称而不是强制实体 ID。
   - 理由：首版输入要轻，用户可以先写角色、势力或地点名称，不需要先完成复杂实体选择器。
   - 替代方案：关系两端必须选择已有种子 ID。缺点是交互更重，且会阻碍快速搭建世界雏形。

## Risks / Trade-offs

- [Risk] localStorage 容量和隐私边界有限，用户数据只存在当前浏览器。→ Mitigation：在 UI 中保持本地草稿定位，不承诺跨设备同步；payload 只保存文本字段和少量数字。
- [Risk] 损坏或旧版本本地数据可能导致工作台渲染失败。→ Mitigation：解析函数必须做字段类型校验，失败时返回 `null` 并使用默认 mock。
- [Risk] 关系参与方名称自由输入可能出现不存在或重名。→ Mitigation：首版允许自由文本，同时在展示层以名称为准；后续接后端或实体图谱时再做 ID 化迁移。
- [Risk] 用户误以为保存种子会启动世界运行。→ Mitigation：不改写运行状态、事件、章节、秘密和趋势；按钮文案和模块标题避免暗示 Agent 已运行。
- [Risk] 表单字段较多，窄屏容易拥挤。→ Mitigation：复用现有 overlay/panel 响应式模式，四类种子用分区或标签切换，长文本字段固定最小高度并允许滚动。
- [Risk] 安全上 localStorage 文本可被同源脚本读取。→ Mitigation：不存储敏感凭据，不执行用户输入，不使用 `dangerouslySetInnerHTML`，展示文本按 React 默认转义渲染。
- [Risk] 每次输入都写 localStorage 可能影响性能。→ Mitigation：采用显式保存或局部状态编辑后批量保存；派生统计使用 `useMemo`。

## Migration Plan

1. 新增类型和本地存储模块，定义 `WorldSeedAssets`、四类 seed item、默认空状态、解析保存和订阅函数。
2. 在工作台中读取种子快照，并派生统计、角色和关系展示数据。
3. 增加内容种子编辑入口和面板，支持四类种子的新增、编辑、删除和保存。
4. 保持无种子或损坏数据时回退到现有 `dashboardData`。
5. 增加前端静态检查和 Playwright 验收，覆盖保存、刷新恢复和工作台联动。

回滚策略：删除新增种子模块、类型和编辑面板，恢复工作台模块直接读取 `dashboardData`；保留浏览器中的种子 localStorage 数据但停止读取，不影响现有创建世界和世界设定功能。

## Open Questions

- 角色种子的“状态”是否需要预设枚举，还是首版保持自由文本。
- 关系紧张度是否用 0-100 数值滑块，还是用低/中/高并映射到数值展示。
- 内容种子入口放在顶部“资源箱”按钮、侧栏“角色/关系”项，还是世界设定面板内的独立标签。
