## Context

当前前端已经有根路径水墨工作台和 `/events` 事件日志页，工作台中存在“最新故事章节”展示，但还没有集中浏览故事章节的独立页面。用户提出的首版目标是展示已生成的故事内容，并允许第一版使用 mock 故事章节。

受影响文件和模块：
- 新增 `src/app/stories/page.tsx`：故事章节列表页路由。
- 可新增 `src/app/stories/[chapterId]/page.tsx`：阅读按钮进入的 mock 章节阅读占位页；若实现时选择列表内展开，也应保持同等可访问阅读行为。
- 新增 `src/components/stories/**` 或等价组件：章节列表、章节卡片、空状态和阅读视图。
- 可新增 `src/lib/story-chapters.ts` 或组件内常量：集中保存 mock 章节数据，字段包含标题、生成时间、世界时间、主要角色、摘要和正文占位。
- `src/app/globals.css`：补充故事章节列表页样式，沿用现有宣纸、墨色、细线卡片和低饱和界面语言。
- `src/components/dashboard/ink-dashboard.tsx`：仅在实现阶段需要入口时，允许把工作台“继续阅读”或章节入口链接到 `/stories`，但本变更不要求修改工作台既有展示契约。

约束：
- 首版只使用前端 mock 章节，不调用 FastAPI、数据库、LangGraph 或任何 Agent 流程。
- 不读取或写入本地 world runtime 状态，避免和后续真实章节持久化方案耦合。
- 页面是应用内工作界面，优先保证章节扫描效率、按钮可点击和响应式可读性。

## Goals / Non-Goals

**Goals:**
- 提供 `/stories` 故事章节列表页，集中展示 mock 章节。
- 每个章节条目展示章节标题、生成时间、对应世界时间、主要角色、摘要和阅读按钮。
- 让阅读按钮进入可读的章节内容占位，而不是无效链接。
- 保持桌面端和窄屏端的布局可读，长标题、长角色名和长摘要不得撑破容器。
- 保持首版本地静态边界，为后续接入真实故事生成结果预留替换点。

**Non-Goals:**
- 不实现真实故事生成、重新生成、发布、编辑、删除、筛选、搜索或分页。
- 不新增后端 API、数据库表、pgvector 检索或远程同步。
- 不触发世界推进、角色行动、事件总结或章节生成 Agent。
- 不设计最终的章节数据模型迁移；mock 字段只服务首版页面验证。

## Decisions

1. 首版使用 `/stories` 作为章节列表路由。
   - 方案：新增 `/stories`，页面内部直接加载 mock 章节数组。
   - 理由：当前应用已有 `/events` 这种顶层功能页，且还没有 `/worlds/[worldId]` 详情路由；使用顶层 `/stories` 可以在短时间内完成可访问页面。
   - 替代方案：使用 `/worlds/[worldId]/stories`。优点是世界归属更明确，缺点是需要先建立世界详情路由和参数校验，超出首版范围。

2. mock 章节数据集中定义为类型化数组，而不是散落在 JSX 中。
   - 方案：定义 `StoryChapter` 类型和 `mockStoryChapters` 数据，字段包含 `id`、`title`、`generatedAt`、`worldTime`、`mainCharacters`、`summary`、`content`。
   - 理由：后续接入真实数据时可以替换数据来源，并让列表页和阅读页复用同一份 mock 数据。
   - 替代方案：在卡片 JSX 中硬编码文本。优点是最快，缺点是阅读页复用困难，也更容易遗漏字段。

3. 阅读按钮首版使用详情页或等价详情状态承接。
   - 方案：优先新增 `/stories/[chapterId]` mock 阅读页，按钮使用 Next.js `Link` 导航；如果实现时为了压缩工期选择列表内展开，也必须展示章节标题、世界时间、主要角色和正文占位。
   - 理由：用户明确需要“阅读按钮”，按钮应有实际可验证行为；详情页也更接近后续真实章节阅读体验。
   - 替代方案：按钮只显示 disabled 或 toast。优点是实现更少，缺点是不可验证“阅读”工作流，用户体验不足。

4. 列表布局采用高密度章节卡片，不做营销式落地页。
   - 方案：页面顶部提供标题、当前 mock 状态和章节数量；主体为章节卡片列表或两列网格，每张卡片固定展示元信息和摘要。
   - 理由：这是叙事工作台的内容浏览页面，用户需要扫描和比较章节，而不是被大幅 hero 或装饰性卡片干扰。
   - 替代方案：大图 hero 加营销文案。优点是视觉冲击强，缺点是不符合应用内工具页的信息密度。

5. 首版保持纯前端静态，不接入 runtime state。
   - 方案：页面只读取 mock 数据，避免读写 localStorage。
   - 理由：真实章节未来可能来自事件日志整理、Agent 输出或后端持久化；现在提前写入本地数据结构会增加迁移成本。
   - 替代方案：把 mock 章节写入 `WorldRuntimeState`。优点是能模拟真实 active world 章节，缺点是会污染当前运行状态契约。

## Risks / Trade-offs

- [Risk] 使用 mock 数据会让用户误以为已有真实生成能力。→ Mitigation：页面用清晰但不突兀的状态文案标记为“示例章节”或“mock 内容”，并避免触发生成相关操作。
- [Risk] `/stories` 没有 worldId，后续多世界章节归属需要迁移。→ Mitigation：mock 数据包含世界时间和可选世界名字段；后续建立世界路由后再补 `/worlds/[worldId]/stories`，保留 `/stories` 作为 active world 快捷页或重定向。
- [Risk] 章节摘要或角色列表过长导致卡片溢出。→ Mitigation：实现时限制摘要行数、角色标签换行，并在移动端改为单列布局。
- [Risk] 阅读详情页复制列表页数据查找逻辑。→ Mitigation：集中导出 mock 数据和查找 helper，列表和详情共用。
- [Risk] 用户输入或后续真实正文可能包含 HTML。→ Mitigation：React 默认按文本渲染；实现时不得使用 `dangerouslySetInnerHTML`。
- [Risk] 纯静态页面不覆盖真实数据接入问题。→ Mitigation：本次目标限定为首版展示；任务中保留类型化字段和替换点，后续单独提出真实章节持久化变更。

## Migration Plan

1. 新增类型化 mock 章节数据，保证每条数据包含标题、生成时间、世界时间、主要角色、摘要和正文占位。
2. 新增 `/stories` 列表页和章节列表组件，展示全部 mock 章节。
3. 实现阅读按钮，跳转到 mock 详情页或打开等价阅读视图。
4. 按需新增工作台入口链接，但不改变工作台原有数据读取行为。
5. 运行 TypeScript/lint 检查，并用浏览器验证桌面和窄屏下列表、按钮和阅读视图均可读。

回滚策略：
- 删除 `/stories` 和可选 `/stories/[chapterId]` 路由、故事组件、mock 数据和样式。
- 如新增了工作台入口链接，恢复为原有链接或占位。
- 因为首版不写入本地数据、不新增接口，回滚不需要数据迁移。

## Open Questions

- 阅读按钮首版是否必须进入独立详情页，还是列表内展开即可？建议实现独立详情页，便于后续接真实正文。
- mock 章节是否需要绑定 active world 名称？建议首版使用固定 mock 世界信息，后续真实数据接入时再绑定 active world。
- 后续真实章节生成结果由前端本地 runtime、后端 API 还是 Agent 输出表承接？当前不阻塞首版列表页。
