## Context

当前前端已经有本地世界库、active world、按 `worldId` 隔离的世界运行记录，以及工作台内的“近期事件”模块。`WorldRuntimeState.events` 目前只保存事件标题、摘要、类型、发生日期和创建时间；工作台最多把事件映射成近期摘要列表，无法让用户完整追溯世界历史。

受影响文件和模块：
- `src/types/world-runtime.ts`：扩展事件字段和事件重要性类型。
- `src/lib/world-runtime.ts`：扩展事件解析、创建、排序和本地状态兼容逻辑。
- `src/components/dashboard/ink-dashboard.tsx`：更新“查看全部”或事件入口跳转，并让轻量记录入口与扩展字段兼容。
- 新增事件日志页面路由和组件，例如 `src/app/events/page.tsx`、`src/components/events/**` 或等价组织。
- `src/app/globals.css`：补充页面布局、事件列表和详情面板样式。
- `README.md` 或相关 OpenSpec 文档：如实现阶段需要，可补充事件日志入口和本地状态说明。

约束：
- 首版仍是前端本地能力，读取 active world 的 `WorldRuntimeState`，不调用 FastAPI、数据库或 Agent。
- 旧 localStorage 事件数据必须继续可读。
- 页面需要延续现有水墨工作台视觉语言，但事件日志是工作界面，应优先保证扫描、比较和追溯效率。

## Goals / Non-Goals

**Goals:**
- 提供完整事件日志页面，让用户按时间倒序查看 active world 的所有事件。
- 在列表层展示时间、标题、参与角色、地点、影响和摘要。
- 在详情层展示事件详情、类型、重要性、参与角色、地点和影响。
- 扩展事件记录结构，让事件成为后续故事生成可用的事实底稿。
- 保持本地运行记录、世界切换、旧事件兼容和工作台近期事件展示稳定。

**Non-Goals:**
- 不实现后端事件表、数据库迁移、API 路由或远程同步。
- 不触发 LangGraph、角色行动、自动世界推进或叙事生成。
- 不实现复杂搜索、分页、批量编辑、事件关系图或时间线可视化。
- 不把事件详情写成小说正文；事件日志记录事实，不负责文学化表达。

## Decisions

1. 首版事件日志使用 `/events` 读取 active world，而不是新增世界 ID 路由。
   - 方案：新增 `/events` 页面，页面内部通过 `world-library` 读取 active world，再用 active `worldId` 读取运行记录。
   - 理由：当前应用还没有 `/worlds/[id]` 详情路由，active world 已经是工作台状态的权威来源；使用 `/events` 可以在 2-3 小时范围内完成闭环。
   - 替代方案：直接设计 `/worlds/[worldId]/events`。优点是 URL 更明确，缺点是需要先建立世界详情路由和 worldId 参数校验，超出本次范围。

2. 事件扩展字段直接进入 `WorldRuntimeEvent`，而不是新增独立 `EventLogEntry`。
   - 方案：在现有事件类型上增加 `detail`、`participants`、`location`、`impact`、`importance` 等字段，并让解析函数为旧事件提供默认值。
   - 理由：工作台近期事件、统计项和新页面都读取同一事件源，避免出现摘要事件和日志事件两套数据不同步。
   - 替代方案：新增独立日志集合。优点是页面模型更自由，缺点是需要同步创建、迁移和统计，当前阶段成本更高。

3. 事件列表默认用现有倒序 helper，排序优先 `createdAt`，其次用 `date` 可读回退。
   - 方案：继续复用或扩展 `getRecentWorldRuntimeEvents`，保证新增页面和工作台排序一致。
   - 理由：当前世界日期可能是中文纪年文本，不能可靠做时间计算；`createdAt` 是稳定 ISO 字符串，更适合本地排序。
   - 替代方案：解析中文世界时间。优点是更贴近叙事时间，缺点是历法规则不完整，容易制造错误顺序。

4. 页面采用列表加详情的工作台布局。
   - 方案：桌面端左侧为事件列表，右侧为选中事件详情；窄屏端详情下移或使用展开式布局。
   - 理由：事件日志需要高密度扫描，列表加详情比卡片瀑布更适合对比时间、角色、地点和影响。
   - 替代方案：纯卡片列表。优点是实现简单，缺点是长详情会降低扫描效率。

5. 手动记录事件入口保持轻量，不强制所有扩展字段必填。
   - 方案：标题和摘要继续必填；参与角色、地点、影响、详情和重要性可以为空或使用默认值。
   - 理由：不能让记录事件的成本过高；后续故事生成需要更丰富字段，但首版应允许用户逐步补全。
   - 替代方案：完整表单强校验。优点是数据质量更高，缺点是当前手动维护阶段会显著增加摩擦。

## Risks / Trade-offs

- [Risk] 旧事件没有参与角色、地点、影响或详情，导致页面信息稀疏。→ Mitigation：解析层补默认字段，页面用“未记录地点”“暂无影响记录”或摘要回退详情。
- [Risk] 中文世界时间无法可靠排序。→ Mitigation：首版以 `createdAt` 为主排序，并在页面展示世界时间文本；未来如引入标准世界时间戳再切换排序依据。
- [Risk] 事件字段扩展后记录表单变复杂。→ Mitigation：工作台仍保留轻量输入；完整页面可以提供更宽松的补充字段，但只要求标题和摘要。
- [Risk] 页面和工作台同时渲染事件时逻辑重复。→ Mitigation：将排序、解析、字段回退集中在 `world-runtime` helper 或事件组件边界。
- [Risk] 本地事件数量增长后渲染变慢。→ Mitigation：首版本地数据规模有限；实现时避免昂贵计算，排序使用 memo；后续再考虑分页或虚拟列表。
- [Risk] 事件详情可能含敏感或不适合展示的用户输入。→ Mitigation：React 默认转义文本；实现时不得使用 `dangerouslySetInnerHTML` 渲染事件内容。

## Migration Plan

1. 扩展 `WorldRuntimeEvent` 类型与解析函数，确保旧事件缺少新增字段时仍能返回规范化事件。
2. 更新事件创建逻辑，让新增字段可选写入；旧工作台记录事件入口未提供字段时使用默认值。
3. 新增 `/events` 页面并读取 active world 的 runtime events。
4. 将工作台“近期事件”的查看全部或事件导航入口连接到 `/events`。
5. 运行 TypeScript/lint 检查，并用浏览器验证桌面和窄屏页面不会空白、重叠或丢失主要信息。

回滚策略：
- 删除 `/events` 页面和事件入口链接，恢复工作台仅展示近期事件摘要。
- 保留已经写入 localStorage 的扩展字段；旧逻辑只读取既有字段，不需要清理用户数据。

## Open Questions

- 事件重要性首版是否只需要 `普通 / 重要 / 转折` 三档，还是需要额外的“伏笔”“结局”标签？建议首版保持三档。
- 参与角色首版使用自由文本数组，还是必须从角色种子中选择？建议首版允许自由文本，并在 UI 中提示可填多个角色。
- 后续世界路由建立后，是否将 `/events` 迁移或补充为 `/worlds/[worldId]/events`？当前不阻塞实现。
