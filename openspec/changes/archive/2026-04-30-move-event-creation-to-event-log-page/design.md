## Context

根路径工作台已经通过 `src/components/dashboard/ink-dashboard.tsx` 展示“近期事件”，并在同一模块中内嵌完整记录事件表单。事件日志页 `src/components/events/world-event-log.tsx` 已经能够读取 active world 的本地运行记录、倒序展示完整事件列表并显示事件详情。

现有事件数据契约位于 `src/types/world-runtime.ts` 和 `src/lib/world-runtime.ts`，已经支持标题、摘要、类型、参与角色、地点、影响、详情和重要性字段，也已经具备 `appendWorldRuntimeEvent`、`getRecentWorldRuntimeEvents` 和 localStorage 订阅刷新机制。本变更不需要新增数据字段，只需要重新分配页面职责。

受影响模块：
- `src/components/dashboard/ink-dashboard.tsx`
- `src/components/events/world-event-log.tsx`
- `src/app/globals.css`
- `openspec/specs/ink-dashboard-ui/spec.md`
- `openspec/specs/world-event-log-page/spec.md`
- Playwright 验证脚本或截图输出

## Goals / Non-Goals

**Goals:**
- 让工作台“近期事件”成为只读预览模块，最多展示少量最新事件，并保持页面高度稳定。
- 将手动记录事件入口移动到事件日志页，使事件页同时支持完整浏览、详情查看和新增维护。
- 在事件日志页提供可控滚动窗口，避免大量事件或较长表单让页面无限变长。
- 复用现有本地世界运行记录逻辑，保存后立即刷新事件列表，并默认展示或选中新创建的事件。
- 保持桌面和窄屏端均可使用，不出现文本重叠、横向溢出或主要操作不可达。

**Non-Goals:**
- 不新增后端 API、数据库持久化或远程同步。
- 不触发 LangGraph、角色行动、叙事生成或自动事件总结。
- 不修改 `WorldRuntimeEvent` 字段、localStorage key 或存储版本。
- 不实现事件编辑、删除、筛选、搜索或批量导入。
- 不改变“运行一天”结果面板和章节列表页面行为。

## Decisions

1. 工作台只保留近期事件预览，不保留录入表单。

   理由：工作台是世界状态总览，内嵌完整事件表单会与统计、运行状态、故事章节和角色模块竞争视觉权重。只读预览能更好地保持信息密度和版面节奏。

   备选方案：保留一个折叠表单或轻量弹层。该方案仍会让工作台承担维护职责，用户需要在两个页面理解同一动作，不利于后续扩展事件字段。

2. 事件日志页接管“记录事件”入口，并复用 `appendWorldRuntimeEvent`。

   理由：事件页已经读取 active world、展示完整列表和详情，是维护事件事实底稿的自然位置。复用现有 helper 可以保持必要字段校验、结构化字段默认值、排序和订阅刷新行为一致。

   备选方案：新建单独 `/events/new` 页面。该方案路由清晰，但首版会增加页面跳转和状态回填成本，也不利于用户创建后立即在列表中核对事件。

3. 桌面端采用“详情/新建”右侧面板切换，移动端使用同一内容流中的可滚动表单区域。

   理由：现有事件页已经是列表 + 详情的双栏结构。把新建状态放入详情栏，可以避免第三栏或额外弹窗，并让用户在同一上下文中创建后查看结果。窄屏端继续单列堆叠，保证列表和表单都可滚动访问。

   备选方案：使用 modal。该方案实现简单，但长表单在移动端容易产生遮挡和滚动焦点问题，也会割裂与事件列表的关系。

4. 事件数量增长时限制容器高度，而不是限制事件页数据读取数量。

   理由：事件页的职责是完整账本，必须展示所有可解析事件；视觉上通过 `.event-log-list` 和详情/表单区域内部滚动控制页面高度。工作台才限制展示数量。

   备选方案：事件页分页或只展示最近 N 条。该方案适合后续大量数据阶段，但当前本地首版没有筛选和分页状态需求，会增加实现面。

5. 无事件状态的主行动指向事件页创建，工作台不直接创建第一条事件。

   理由：保持一致的职责边界。用户在世界页看到“暂无事件”时，可以进入事件日志页创建；事件页空状态则提供“记录第一条事件”的主要入口。

   备选方案：工作台空状态直接放一个小型记录入口。该方案会重新引入双入口维护问题。

## Risks / Trade-offs

- [Risk] 用户习惯从工作台直接记录事件，入口移动后可能需要额外一次点击 → Mitigation：工作台近期事件模块保留清晰的“查看全部”或“记录事件”链接，直接进入事件日志页。
- [Risk] 事件页新增表单后与详情面板互相抢占空间 → Mitigation：使用明确的模式切换，详情模式显示选中事件，新建模式显示表单，并为面板设置最大高度和内部滚动。
- [Risk] 保存后选中逻辑可能因为事件排序或 React 状态更新显示旧详情 → Mitigation：保存成功后根据新增事件 ID 更新选中状态，或在保存后依赖刷新后的首条最新事件作为默认详情。
- [Risk] 删除工作台表单后遗留 CSS 造成无用样式或视觉回归 → Mitigation：清理工作台专用 `.event-entry` 样式，或迁移为事件页表单样式并通过 Playwright 截图验证。
- [Risk] 无 active world 时事件页表单不可保存 → Mitigation：无 active world 时禁用创建入口或显示只读空状态，不调用保存逻辑。

## Migration Plan

1. 更新工作台组件，移除事件表单状态、提交处理和 `onRecordEvent` 属性，只保留只读近期事件列表与事件页入口。
2. 在事件日志页增加记录事件模式、表单状态和提交逻辑，复用 `appendWorldRuntimeEvent` 与 `saveWorldRuntimeState`。
3. 调整事件页空状态：有 active world 且无事件时允许进入记录事件模式；无 active world 时继续展示可读回退。
4. 调整样式，让工作台近期事件高度稳定，让事件页列表和新建/详情面板具备内部滚动。
5. 更新或新增 Playwright 验证，覆盖桌面和移动端：工作台无录入表单、事件页可创建事件、大量事件列表可滚动。
6. 回滚时恢复工作台表单和 `onRecordEvent`，移除事件页创建模式；本地事件数据无需迁移。

## Open Questions

- 事件页新建模式是否需要通过 query 参数（例如 `/events?mode=new`）深链打开，还是只在页面内按钮切换。
